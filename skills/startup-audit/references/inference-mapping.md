# Inference Mapping

Stage 1b. Turn the extracted signal set into an inferred business model â€” tiered
by confidence, pinned to provenance, and diffed against the claimed story.

**Keystone:** infer **into `validation-canvas`'s nine Lean Canvas blocks**. Don't
invent a framework. This reuses the artifact the rest of the pipeline speaks,
auto-generates the confidence map (code evidences some blocks, not others), and
makes the un-evidenced blocks the `riskiest-assumption-test` handoff.

---

## The confidence contract (derive, never judge)

The tier is a **function of the provenance**, not an opinion:

| Tier | Rule | Example |
|---|---|---|
| **observed** | A deterministic signal is directly present. | `invoice.paid` webhook handler + populated plan table â†’ Revenue Streams = observed |
| **inferred** | Reasoned from a pattern; the signal implies but doesn't prove. | Stripe dep + `Subscription` entity (no webhook wired) â†’ Revenue Streams = inferred |
| **unknown** | No signal. Block renders empty. | No README problem statement, nothing in code â†’ Problem = unknown |

**Hard gate: `provenance == null` â†’ the field cannot render as a claim.** Unknowns
are never filled by reasoning about the market. A block with no signal stays
empty and becomes a test hypothesis, not a guess.

## Signal â†’ Lean Canvas block mapping

This table is also the confidence map: blocks at the top read strongly from code;
blocks at the bottom are almost always `unknown` from code alone.

| Lean Canvas block | Primary code signal | Typical tier |
|---|---|---|
| **Solution** | Routes, modules, built features, schema breadth, package `description` | **observed** (it's literally the code) |
| **Customer Segments** | Auth/tenancy model â€” `Org/Team/Seat`â†’B2B, single `users`â†’B2C, public APIâ†’developer | observed â†’ inferred |
| **Revenue Streams** | Payment SDK + plan tables + `invoice.paid` webhook + metering; `free`/`pro` enums; `STRIPE_PRICE_ID` count | observed (money code wired) â†’ inferred (dep only) |
| **Key Metrics** | Analytics instrumentation â€” *what they track reveals what they value* | inferred |
| **Channels** | Marketing pages, SEO, email/CRM SDKs, referral/invite tables, featured/discovery surfaces | inferred |
| **Cost Structure** | Infra (cloud/k8s/IaC), datastore, LLM token usage logs, third-party APIs | inferred |

| **Unique Value Proposition** | README claims; not code-evidenceable | **unknown** â†’ RAT handoff |
| **Unfair Advantage** | Essentially never in code | **unknown** â†’ RAT handoff |

`Revenue model` specifics (per-seat / usage / flat / take-rate / freemium) live
inside the Revenue Streams block â€” they are canvas content, **not** a
team-composer panel signal (Phase 3 only pre-fills team-selection signals).

## Build-vs-claim diff (the headline output)

Contrast the inferred (coded) model against the **claimed** story (README,
marketing copy, live URL). This is what makes the output *diligence* rather than
code archaeology. Render as two columns: **coded reality | claimed story**, with
the delta called out.

**Run it bidirectionally:**

- **claimed > built** â€” the deck/README sells something the code doesn't have
  (e.g. "enterprise, usage-based" but one flat price and no metering; "for
  organizations" but no tenancy).
- **built > claimed** â€” substantial built capability the story under-sells (e.g.
  a full AI subsystem barely mentioned in the value prop).

**Parse README roadmap markers** ("Future", "Coming soon", "Roadmap", "Planned",
"WIP"). A capability the tagline sells but the roadmap marks "Future" is a diff
finding â€” the product promises what it hasn't shipped.

Each diff row carries a `finding-id` (e.g. `F1`, `F2`) so the Options tier
(Phase 3) can cite it.
