# i18n & Localization

> **Rule:** Localization is a **Day 1** decision. Set it up when the project is created — not as a retrofit.

> **Skill:** When working on translations, load [`skills/i18n-contextual-rewriting/SKILL.md`](../../skills/i18n-contextual-rewriting/SKILL.md) for language-specific role teams and cultural review processes.

---

## Why Day 1

Retrofitting i18n onto an existing codebase is one of the most tedious refactors in frontend development. Every hard-coded string must be found, extracted, keyed, and translated. Starting with i18n from the beginning costs almost nothing extra per feature.

---

## Core Rules

1. **No hard-coded user-facing text.** Every string shown to users must come from a translation file.
2. **Every change that touches UI text must update ALL translation files.** If you add or modify a key in one locale, add it to every other locale (use the default language value as a placeholder if the translation isn't available yet).
3. **Translation files are the source of truth** for all user-facing copy — labels, buttons, messages, errors, tooltips, placeholders, page titles.
4. **Keys use dot-notation namespacing** scoped to feature or page (e.g., `auth.login.title`, `dashboard.stats.totalUsers`).

---

## File Structure

```
locales/
├── en.json          # Default / source language
├── th.json          # Thai
├── ja.json          # Japanese
└── ...              # One file per supported locale
```

- **Location:** `locales/` in the app root (or `src/locales/` if framework dictates).
- **Naming:** Use [IETF language tags](https://en.wikipedia.org/wiki/IETF_language_tag) (`en`, `th`, `ja`, `zh-TW`).

### Supported Formats

| Format | Example | Notes |
|--------|---------|-------|
| JSON | `en.json`, `messages.json` | Most common. Flat or nested key-value. |
| YAML | `en.yml`, `messages.yaml` | Common in Rails, Spring, etc. |
| TypeScript | `en.ts`, `translations.ts` | Used when types are shared. Often exports an object `as const` or uses `satisfies`. |
| JS/MJS | `en.js`, `en.mjs` | Legacy or framework-specific. |

---

## Key Format

```json
{
  "auth": {
    "login": {
      "title": "Sign In",
      "subtitle": "Welcome back",
      "emailLabel": "Email address",
      "passwordLabel": "Password",
      "submitButton": "Sign In",
      "forgotPassword": "Forgot your password?"
    }
  },
  "common": {
    "save": "Save",
    "cancel": "Cancel",
    "loading": "Loading…",
    "error": "Something went wrong"
  }
}
```

### Key Naming Conventions

| Pattern | Example | Use |
|---------|---------|-----|
| `feature.page.element` | `auth.login.title` | Page-specific text |
| `common.action` | `common.save` | Shared across pages |
| `error.code` | `error.notFound` | Error messages |
| `validation.rule` | `validation.required` | Form validation |

## Recommended Libraries

| Stack | Library | Notes |
|-------|---------|-------|
| **SvelteKit** | [svelte-i18n](https://github.com/kaisermann/svelte-i18n) or [Paraglide](https://inlang.com/m/gerre34r/library-inlang-paraglideJs) | Paraglide is tree-shakeable and type-safe |
| **Next.js** | [next-intl](https://next-intl-docs.vercel.app/) | Best DX for App Router |
| **React (general)** | [react-i18next](https://react.i18next.com/) | Most popular, mature |
| **Expo / React Native** | [expo-localization](https://docs.expo.dev/versions/latest/sdk/localization/) + react-i18next | Detect device locale + translate |
| **Backend / API** | Store locale in user profile or `Accept-Language` header | Return localized error messages |

## Agent Workflow

When working on any task that adds or modifies user-facing text:

1. **Check if i18n is set up.** If not, set it up before writing any UI text.
2. **Never hard-code strings.** Use the i18n library's `t()` function (or equivalent).
3. **Add keys to the default locale file** (`en.json`).
4. **Copy the new keys to every other locale file** with the default language value as placeholder.
5. **Log in `memory.log`** which translation keys were added or changed.

### When Setting Up a New Project

During the **Setup** phase, include an i18n setup task:

```yaml
task:
  id: "setup-i18n"
  name: "Configure internationalization"
  phase: "setup"
  milestone: "setup-monorepo"
  area: "shared"
  depends_on: ["setup-scaffold-web"]
  complexity: 3
  description: |
    Install i18n library, create locales/ directory,
    add default locale file, configure locale detection.
  acceptance_criteria:
    - "i18n library installed and configured"
    - "locales/en.json exists with sample keys"
    - "t() function works in at least one component"
```

## Editing Translation Files

Translation files can grow to thousands of keys. **Never rewrite an entire translation file** — this will exceed output token limits and cause errors. Always use surgical, minimal edits.

> [!CAUTION]
> Do NOT output the full file content or use a "replace entire file" operation on translation files. If your tooling only supports full-file writes, use the script approach below.

### Surgical Edits (Default Approach)

When adding, updating, or removing keys, target **only the affected lines**.

**JSON example:**

```diff
   "dashboard.title": "Dashboard",
+  "dashboard.subtitle": "Welcome back",
+  "dashboard.empty_state": "No data available",
   "dashboard.footer": "© 2025"
```

**TypeScript example:**

```diff
 export const en = {
   dashboard: {
     title: 'Dashboard',
+    subtitle: 'Welcome back',
+    emptyState: 'No data available',
     footer: '© 2025',
   },
 } as const;
```

### Script Approach (Bulk Operations)

When changes are too numerous for surgical edits (e.g., 50+ keys, restructuring namespaces, syncing across locales), write a **one-off script**.

**For JSON files:**

```ts
import fs from 'fs';

const filePath = './locales/en.json';
const translations = JSON.parse(fs.readFileSync(filePath, 'utf-8'));

const newKeys = {
  'feature.title': 'New Feature',
  'feature.description': 'This is a new feature',
};

Object.assign(translations, newKeys);
fs.writeFileSync(filePath, JSON.stringify(translations, null, 2) + '\n');
```

**For TypeScript files** — choose a strategy:

| Strategy | When to Use |
|----------|-------------|
| **Dynamic import** | Structure is `export default` or named export; `as const` / `satisfies` can be re-applied |
| **AST manipulation** (`ts-morph`) | Type safety must be preserved exactly |
| **Regex / text insertion** | Simple cases — appending a few keys to a well-known structure |

> When writing back a TS file, preserve the original structure (`as const`, `satisfies`, imports).

### Syncing Across Locales

When adding keys to all locale files:

1. Define the new keys in a single object.
2. Write a script that iterates over all locale files.
3. For each file: read → merge (without overwriting existing keys) → write.
4. Log which files were updated and which keys were added.

> Do NOT manually edit each locale file one by one — this is error-prone and token-expensive.

### Validate After Changes

After any translation file modification:

- Verify the file is valid (parseable JSON, valid TS syntax, etc.)
- Run any existing i18n linting or type-checking commands if available
- Confirm no existing keys were accidentally removed or overwritten

---

## Checklist for Text Changes

- [ ] New/modified text uses `t('key')` — not a raw string
- [ ] Key added to default locale file
- [ ] Key added to **all** other locale files (placeholder OK)
- [ ] Key follows dot-notation naming convention
- [ ] No orphaned keys left behind (if replacing text, remove old keys)
- [ ] Translation file is valid after edit (parseable, no missing keys)
- [ ] Used surgical edits — not a full file rewrite
